# Z-Image-Turbo: Proper 4-way Tensor Parallel + Codegen

Runs [Tongyi-MAI/Z-Image-Turbo](https://huggingface.co/Tongyi-MAI/Z-Image-Turbo)
transformer on 4 Tenstorrent devices with full tensor parallelism and emits
Python codegen (EmitPy) for the compiled graph.

## What it does

**Head padding** — The transformer has 30 attention heads, which doesn't divide
evenly by 4. Two dummy zero-weight heads are appended to all `to_q/k/v/out`
projections to get 32 heads (8 per device). This is mathematically transparent:
dummy heads produce Q=K=V=0, so their attention output is 0 and they contribute
nothing to `to_out`. The script verifies this with a CPU bit-exact check
(PCC=1.0, max_diff=0.0) before running on TT.

**Sharding** — Standard Megatron-style TP on a `(1, 4)` mesh:
- `to_q/k/v`, `w1/w3` → column-parallel `("model", None)`
- `to_out`, `w2` → row-parallel `(None, "model")`

**Codegen** — `backend="codegen_py"` in compile options causes the TT compiler
to emit Python (EmitPy) instead of executing. Generated files land in
`codegen_output/transformer/`.

**Text encoder** — Skipped. The Qwen3 text encoder hits a `ttnn.cumsum` u8
limitation in `transformers/masking_utils.py`. Dummy `cap_feats [7, 2560]` are
used instead, matching the CPU reference inputs so the transformer comparison
remains valid.

## Requirements

- 4 Tenstorrent devices (e.g. TT-QuietBox / 4× Wormhole n150)
- tt-xla environment activated (`source venv/activate` from repo root)
- Python packages: `diffusers`, `transformers`, `torch`, `torch_xla`
  (all present in the tt-xla venv)
- HuggingFace model weights: `Tongyi-MAI/Z-Image-Turbo` (downloaded automatically
  on first run; set `HF_TOKEN` env var if you hit rate limits)

## Running

From the repo root:
```bash
source venv/activate
python z_image_turbo/proper_tp/run.py              # codegen mode (default)
python z_image_turbo/proper_tp/run.py --validate   # hardware execution + CPU comparison
```

Or from within this directory (after activating the venv from repo root):
```bash
python run.py            # codegen
python run.py --validate # validate
```

For quick iteration on codegen without running CPU checks:
```bash
python codegen_debug.py             # with tt_legacy_compile
python codegen_debug.py --no-legacy # without tt_legacy_compile
```

## Files

| File | Purpose |
|------|---------|
| `run.py` | Main script: CPU validation + codegen (`--codegen`) or hardware run (`--validate`) |
| `common.py` | Self-contained utilities: model loading, mesh setup, sharding, all patches |
| `codegen_debug.py` | Fast debug script — skips CPU checks, useful when iterating on codegen |
| `codegen_output/` | Generated by `run.py` or `codegen_debug.py` — EmitPy files and weight tensors |

## Notes

- The ~0.44 max diff between TT and CPU is expected: bfloat16 partial sums from
  4-way allreduce accumulate in a different order than sequential CPU computation.
  PCC ≈ 0.998 confirms the result is numerically correct.
- `_output_capturing_hooks_installed = True` prevents `transformers` from
  installing `register_forward_hook` on every submodule during dynamo tracing,
  which would otherwise cause one graph break per layer.
- `tt_legacy_compile=True` is required for codegen (enables MetaDataProp).
- Codegen produces 2 output directories (`transformer/` and `transformer_0/`).
  This is a TT backend artifact — there is only 1 dynamo graph (verified via
  `torch._dynamo.explain`). The TT compiler does 2 passes; `transformer_0/` is
  the primary complete graph (single clean output tensor `[16, 1, 32, 32]`).
- The `unpatchify` patch (`_unpatchify_xla` in `common.py`) rewrites the final
  reshape to derive target dimensions from the permuted tensor's own shape
  rather than from the Python `size` list argument. This eliminates a potential
  dynamo graph break at the permute/reshape boundary.
