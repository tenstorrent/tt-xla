# Fix: Wrong Result Order in `stablehlo.composite "tenstorrent.topk"`

## Problem

In `failing.mlir` (line 260), the `stablehlo.composite "tenstorrent.topk"` op had its
result types in the wrong order:

```mlir
%206:2 = stablehlo.composite "tenstorrent.topk" %205 {
    composite_attributes = {dim = -1 : i64, k = 4 : i64, largest = true, sorted = true},
    decomposition = @outlined_composite_tenstorrent.topk.impl
} : (tensor<128x32xbf16>) -> (tensor<128x4xi64>, tensor<128x4xbf16>)
                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                              WRONG: (indices, values)
                              EXPECTED: (values, indices)
```

`torch.topk` returns `(values, indices)`. The `TenstorrentTopKConversionPattern` in
`StableHLOLegalizeCompositePass.cpp` assumes this convention — it assigns `result[0]`
as the values type and `result[1]` as the indices type. With the swapped order, the
downstream `ttir.TopKOp` is created with inverted type assignments, producing wrong
outputs at runtime.

---

## Root Cause

The inversion is introduced by two passes that run sequentially inside the
`stablehlo-pipeline` (see `StableHLOPipelines.cpp` lines 62 and 112):

### Step 1 — `shlo_frontend`: correct decomposition, cross-order return

The `@tenstorrent.topk.impl` function generated by torch_xla has this body
(line 2860 of `failing.log`):

```mlir
func.func private @tenstorrent.topk.impl(%arg0: tensor<128x32xbf16>)
    -> (tensor<128x4xbf16>, tensor<128x4xi64>) {          // correct: (values, indices)
  %0 = stablehlo.iota ...
  %1 = stablehlo.broadcast_in_dim %0 ...
  %2:2 = "stablehlo.sort"(%arg0, %1) ...
      -> (tensor<128x32xbf16>, tensor<128x32xi32>)         // sort.0=values, sort.1=indices
  %3 = stablehlo.slice %2#1 [0:128, 0:4] ...              // indices i32  (block pos 3)
  %4 = stablehlo.convert %3 ...                           // indices i64  (block pos 4) ← escape
  %5 = stablehlo.slice %2#0 [0:128, 0:4] ...              // values  bf16 (block pos 5) ← escape
  return %5, %4 : tensor<128x4xbf16>, tensor<128x4xi64>   // cross-order return: values first
}
```

The body ops are ordered **indices-before-values** in the block, but the `return`
statement **cross-references** them to produce the correct `(values, indices)` output
order.

### Step 2 — `FlattenCompositePass` inlines without preserving output position

`FlattenComposite.cpp:102-116` clones the decomposition body ops in their original
block order into the parent function:

```
cloned block order: iota → broadcast → sort → slice_sort#1 (indices) → convert (indices_i64) → slice_sort#0 (values)
```

`FlattenComposite.cpp:160-168` correctly re-wires downstream uses via the terminator:

```cpp
comp.getResult(0).replaceAllUsesWith(values_bf16);   // correct — values slot
comp.getResult(1).replaceAllUsesWith(indices_i64);   // correct — indices slot
```

**The problem:** no annotation records that `values_bf16` is composite result #0 and
`indices_i64` is result #1.

### Step 3 — `ReoutlineCompositePass` sorts escapes by block position → wrong order

`ReoutlineComposite.cpp` identifies escape values (results used outside the group)
and sorts them with `orderByBlockPos`. Since `convert` (producing `indices_i64`) was
cloned at block position 4 and `slice sort#0` (producing `values_bf16`) at position 5,
block-position sorting yields `(indices_i64, values_bf16)`.

The composite is then created with result types `(i64, bf16)` — **the wrong order**.

---

## Fix

Three files were changed. The core idea is: `FlattenCompositePass` annotates each
escape-producing op with its original composite output position (`reoutline.output_pos`),
and `ReoutlineCompositePass` prefers that annotation over block position when ordering
escapes.

### 1. `third_party/tt-mlir/src/tt-mlir/include/ttmlir/Dialect/StableHLO/Utils/StableHLOUtils.h`

Added a new attribute key constant alongside the existing `reoutline.*` family:

```cpp
// before
inline constexpr llvm::StringLiteral
    kReoutlineArgOperandIndicesAttr("reoutline.arg_operand_indices");

// after
inline constexpr llvm::StringLiteral
    kReoutlineArgOperandIndicesAttr("reoutline.arg_operand_indices");
inline constexpr llvm::StringLiteral
    kReoutlineOutputPosAttr("reoutline.output_pos");
```

### 2. `third_party/tt-mlir/src/tt-mlir/lib/Dialect/StableHLO/Transforms/FlattenComposite.cpp`

In `flattenOneComposite`, after remapping each composite result to its inlined
equivalent, annotate the defining op with its output position:

```cpp
// before (lines 160-168)
for (int64_t i = 0; i < static_cast<int64_t>(comp->getNumResults()); ++i) {
    mlir::Value mapped = mapping.lookupOrNull(ret.getOperand(i));
    if (!mapped) { ... return mlir::failure(); }
    comp.getResult(i).replaceAllUsesWith(mapped);
}

// after
for (int64_t i = 0; i < static_cast<int64_t>(comp->getNumResults()); ++i) {
    mlir::Value mapped = mapping.lookupOrNull(ret.getOperand(i));
    if (!mapped) { ... return mlir::failure(); }
    comp.getResult(i).replaceAllUsesWith(mapped);
    // Annotate the defining op with its original output position so that
    // ReoutlineCompositePass can restore the correct result order regardless of
    // the op's block position.
    if (mlir::Operation *defOp = mapped.getDefiningOp()) {
        defOp->setAttr(utils::kReoutlineOutputPosAttr,
                       builder.getI64IntegerAttr(i));
    }
}
```

### 3. `third_party/tt-mlir/src/tt-mlir/lib/Dialect/StableHLO/Transforms/ReoutlineComposite.cpp`

In `analyzeBoundary`, replace the simple `llvm::sort(escapes, orderByBlockPos)` with a
sort that first checks for `reoutline.output_pos` annotations — falling back to block
position only when the annotation is absent. This mirrors how captures already use
`reoutline.arg_operand_indices` for ordering (lines 120-153):

```cpp
// before (lines 155-156)
escapes.assign(escapeSet.begin(), escapeSet.end());
llvm::sort(escapes, orderByBlockPos);

// after
// Collect reoutline.output_pos annotations set by FlattenCompositePass so we
// can restore the original result order (which may differ from block order
// when the decomposition's return statement uses a cross-order return).
llvm::DenseMap<mlir::Value, int64_t> escapeOutputPos;
for (mlir::Value v : escapeSet) {
    if (mlir::Operation *def = v.getDefiningOp()) {
        if (auto posAttr = def->getAttrOfType<mlir::IntegerAttr>(
                utils::kReoutlineOutputPosAttr)) {
            escapeOutputPos[v] = posAttr.getInt();
        }
    }
}

escapes.assign(escapeSet.begin(), escapeSet.end());
llvm::sort(escapes, [&](mlir::Value a, mlir::Value b) {
    auto itA = escapeOutputPos.find(a);
    auto itB = escapeOutputPos.find(b);
    bool hasA = (itA != escapeOutputPos.end());
    bool hasB = (itB != escapeOutputPos.end());
    if (hasA && hasB) {
        return itA->second < itB->second;
    }
    if (hasA != hasB) {
        return hasA; // known positions before unknowns
    }
    return orderByBlockPos(a, b); // fallback to block order
});
```

## Files Changed

| File | Change |
|------|--------|
| `third_party/tt-mlir/src/tt-mlir/include/ttmlir/Dialect/StableHLO/Utils/StableHLOUtils.h` | Add `kReoutlineOutputPosAttr` constant |
| `third_party/tt-mlir/install/include/ttmlir/Dialect/StableHLO/Utils/StableHLOUtils.h` | Same (installed copy) |
| `third_party/tt-mlir/src/tt-mlir/lib/Dialect/StableHLO/Transforms/FlattenComposite.cpp` | Annotate escape-producing ops with `reoutline.output_pos = i` |
| `third_party/tt-mlir/src/tt-mlir/lib/Dialect/StableHLO/Transforms/ReoutlineComposite.cpp` | Sort escapes by `reoutline.output_pos` when present, fall back to block order |

`StableHLOLegalizeCompositePass.cpp` (`TenstorrentTopKConversionPattern`) does **not**
need to change — it correctly assumes `result[0]`=values once the composite result order
is fixed.

## Verification

```bash
pytest -svv tests/torch/ops/test_composite_ops.py::test_composite_topk_both
pytest -svv tests/torch/ops/test_composite_ops.py::test_patched_topk_values
pytest -svv tests/torch/ops/test_composite_ops.py::test_patched_topk_indices
```

After the fix, the compiled model's `stablehlo.composite "tenstorrent.topk"` should
have result types `(tensor<128x4xbf16>, tensor<128x4xi64>)` — values first, indices
second — matching `torch.topk` convention.

The existing `test_reoutline_composite.mlir` test (single-output gelu) is unaffected
because `orderByBlockPos` is preserved as the fallback for escape values that carry no
`reoutline.output_pos` annotation.
