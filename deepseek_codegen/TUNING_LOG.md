# DeepSeek-V3.2 codegen_py decode tuning log

Drives `tune-emitpy-perf` against `deepseek_codegen/graph_0/main.py` (parity-restored — see `CODEGEN_ISSUES.md`). Target: lower the device-time of one decode step (`decode_1_start` → `decode_1_end` signposts in `main.py:13380`) while holding PCC vs `golden_logits.pt` (see "Harness" section below).

## Harness (PCC correctness signal)

`pcc.py` runs `_main` once and produces two signals:

* **Bootstrap PCC (soft).** Compares each of the 10 returned tensors against `baseline_outputs.pt`. PCC=1.0 means the computation is bit-identical to the saved baseline; useful for catching unintended drift in a refactor, but does NOT validate against the reference model.

* **Golden PCC (hard fail, floor=0.9).** `_main` was patched to additionally return the pre-argmax logits tensor (`ttnn_to_layout_267`) as `out[9]`. The TTNN stack has shape `(32 chips, 32 batch_pad, 129280 vocab)`; chips group as `[0..7] / [8..15] / [16..23] / [24..31]` (4 unique batch shards, replicated 8× by the cluster-axis-1 all_gather), and within each chip the 32 batch rows are tile-padding for a single sequence. `golden_logits.pt` is `(128, 1, 129280)` FP32 where all 128 rows are identical (one batch's logits replicated). The canonical comparison is therefore `chip[0].row[0]` vs `golden[0]`, giving **PCC = 0.921842** on the unmodified baseline (bf16 quantization through the LM head and reduce_scatter). Hard floor is 0.9 ("as good as baseline").

The earlier convention of comparing against a self-bootstrapped baseline with threshold ≥ 0.9 was misleading: PCC = 1.0 vs bootstrap is necessary but not sufficient evidence that an optimization is numerically safe, and a real fidelity-improving patch (e.g. HiFi4 → HiFi2) can legitimately read PCC < 1.0 vs bootstrap while still being ≥ 0.9 vs golden.

## Exit notes (after E9 keep)

Iterations E1–E9 covered both fused-op replacements (E1, E3, E5, E6, E7) and semantic-simplification refactors (E8, E9). The wins that landed (E1, E3, E6, E8, E9) drop decode device-time from the original baseline of 320,030 μs to **309,880 μs** — a saving of **−10,150 μs / -3.17%**. E2/E4/E5/E7 all crashed or regressed and were reverted (full history preserved with revert commits).

By far the biggest gains were the two scatter-coalesce steps (E8 + E9 = -9,693 μs combined). The lesson is that "look for missing fused ops" is often a smaller lever than "look for *semantically redundant* per-step work" — the 130 scatters were each cheap individually, but 126 of them were provably no-ops on the running state and removing them dropped almost an order of magnitude more device time than the fanciest fused-op swap (E6).

The remaining non-MoE inefficiency on decode is concentrated in:

- `SparseMatmulDeviceOperation` — 84% of decode device-time, single-handedly bounding any further speedup. Cannot be attacked without a full MoE rewrite (`ttnn.experimental.moe_compute` or `ttnn.experimental.moe_gpt` + `selective_reduce_combine`), which requires weight repacking (gate+up→`w0_w1`, down→`w2`, likely BFP8), a `GlobalSemaphore` pool, mapping tensors, and replacing both `all_to_all_dispatch`/`all_to_all_combine` and the 3 expert matmuls in one shot. This is a multi-iteration project (estimated 1-2 days of dedicated wiring), not a single experiment iteration.

- `ScatterDeviceOperation` — 130 sequential scatters writing into a shared buffer (8,761 μs). Coalescing into a single scatter would require auditing index/value provenance (most come from `main_const_eval_*` slices) and proving non-overlap. High risk of correctness regressions.

- Distributed RMSNorm chain — 5 sites of `rms_norm_pre_all_gather + slice + reshape + dim=0 all_gather + 7 manual ops`. E7 confirmed that the canonical `ttnn.rms_norm_post_all_gather` does not accept our stats layout; landing it requires either a stats-reshape spike or moving to the sharded `ttnn.fused_rms_minimal` (which needs `LayerNormShardedMultiCoreProgramConfig`, persistent stats buffer, `GlobalSemaphore`, and sharded I/O conversions at every call site). Ceiling is ~250-500 μs.

Recommendation: stop the per-experiment loop here, keep the E1+E3+E6 deltas, and treat MoE compute fusion as a separate scoped project.

### E2 attempt — what went wrong (briefly)

What was attempted is in the ledger row. PCC and tracy could not be measured because the run produced NaN/Inf — bootstrap KV cache `max|Δ|` reached `2.23e38` and the sampled tokens diverged 100%. The golden-PCC printout reported `1.000000` only because `pcc.py`'s correlation formula degenerates to 1 once `inf` enters both candidate and golden via shared `nan_to_num`/std math; this is a known false-positive mode of the harness and not real fidelity.

Hypotheses, in decreasing likelihood, for any retry of this fusion:

1. The pre op output is a tile-wide tensor (`[..., 1, 32]`) where only column 0 holds the real `Σx²` and columns 1–31 are undefined. The post op's row-reduce averages across all 256 cells per row of the gathered stats; if the kernel's scaler tile does not isolate column 0 in our specific config (HiFi4 + fp32_dest_acc + 1d core grid + small `H=32` rows), undefined garbage flows into the variance.
2. Gamma in our codegen is `[1, 896]` FP32 in TILE layout. Validation accepts it (no dtype check for TILE-layout gamma), but the kernel may silently fall back to a `Float16_b` path that misinterprets FP32 tiles. The reference test uses BF16 gamma.
3. `all_gather(dim=3, cluster_axis=1)` may not produce the same layout as the reference test's `ttnn.concat(stats, dim=3)` simulation. Worth checking against an actual multi-device unit test.

Pre-flight before another E2-style attempt: in main.py at one site, replace the post output with a `ttnn.to_torch` dump and verify the values are finite and close to the manual chain's output. Don't attempt at all 5 sites until one site is proven numerically correct.

### Parked (prefill-only — DO NOT try on the decode graph)

These kernels live under `experimental/deepseek_prefill/*` and are written against prefill batch shapes (`seq_len` in the hundreds-to-thousands per token-group). Trying them on a decode step would either fail validation or take a slow path that's strictly worse than the current decode ops. Kept here in case a future prefill tuning pass picks them up.

- `ttnn.routed_expert_ffn` — fused `silu(x@gate) * (x@up) @ down` expert FFN.
- `ttnn.experimental.deepseek_prefill.combine` — specialised MoE combine.
- `ttnn.experimental.deepseek_prefill.dispatch` — specialised MoE dispatch.
- `ttnn.experimental.deepseek_prefill.post_combine_reduce` — `to_layout(TILE) + mul(weights) + sum(experts)` after combine.
- `ttnn.experimental.deepseek_prefill.moe_grouped_topk` — full grouped-topk MoE gate.
- `ttnn.experimental.deepseek_prefill.masked_bincount` / `offset_cumsum` / `extract` / `insert` — side products of the prefill dispatch/combine pipeline.

### Patterns / wiring lessons applicable to multiple decode-tier attempts

1. **L1-sharded outputs require a `sharded_to_interleaved` bridge before the next DRAM-interleaved op** (E5 wiring). When fusing into a kernel that outputs L1 sharded TILE (e.g. `deepseek_moe_post_combine_tilize`), budget for one extra `ttnn.sharded_to_interleaved(..., DRAM, INTERLEAVED)` step or rewrite the downstream op to accept L1 sharded input.
2. **The `nd_shard_spec` shape and grid must satisfy `num_shards_wide * num_shards_high == num_cores`** (one shard per core), and `shard_height == TILE_HEIGHT`, and `shard_width` is a multiple of 32 dividing the tensor's last dim. Compute it from the upstream op's actual padded shape.
3. **CCL fused ops (`fused_rms_minimal`, `selective_reduce_combine`, `moe_compute`, …) take a `GlobalSemaphore`** that must outlive the op call. Single-shot codegen needs a persistent allocation pattern — see how the existing CCL ops (`all_to_all_combine`, `all_gather`, `reduce_scatter`) get away without it (they use `subdevice_id=None` and an internal semaphore allocator). For the fused variants we'll have to add a `ttnn.create_global_semaphore(...)` step in const_eval and pass it through.
4. **`rms_norm_post_all_gather`-style ops need BF16 input + BF16 stats AND a scaler-tile pattern in the stats** (E2 hypothesis 1). Before retrying any post-allgather fusion, dump `to_torch(stats)` at one site and confirm the per-row reduction over the gathered stats tile yields the right scalar.
5. **Blackhole vs Wormhole**: many fused kernels above were authored on Wormhole TG. Most accept a `compute_kernel_config` — try `BlackholeComputeKernelConfig` if `WormholeComputeKernelConfig` is rejected. A genuine WH-only kernel will fail at program-factory dispatch and we should skip it. Test `@pytest.mark.requires_device(["TG"])` markers in the upstream test files for a quick sanity check.
